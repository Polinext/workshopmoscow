Что такое
вайбкодинг?

Как лучше писать

Сильное ТЗ лучше разделить на 2–3 запроса

Почему не одним сообщением? Если отправить слишком длинное ТЗ сразу, Codex может дольше думать, упустить часть деталей или начать делать лишнее. Лучше сначала собрать фундамент, а потом добавлять всё, что находится “сверх фундамента”: авторизацию, оплату, CRM, Google Sheets, дизайн и хостинг.
Запрос 01

Фундамент и первая рабочая версия

Ты — продуктовый дизайнер и full-stack разработчик. Создай полноценный Telegram Mini App [название или тема приложения].

Контекст и цель. Приложение создаётся для [компания / эксперт / сообщество / проект]. Его цель — помочь [кому] [решить задачу или получить результат].

Пользователи и роли. Основные пользователи: [клиенты / сотрудники / участники / администраторы]. Предусмотри роли [перечислить роли].

Главный сценарий. После входа человек должен: [шаг 1] → [шаг 2] → [шаг 3 — конечный результат]. Сделай путь коротким, понятным и без лишних действий.

Экраны и функции. Нужны: [главная], [каталог или список], [карточка объекта], [профиль], [заявки / история / избранное]. Пользователь может [просматривать / искать / создавать заявку / редактировать данные].

Важно: сначала создай простую первую версию без сложных интеграций. Запусти проект, проверь основной сценарий и исправь ошибки.

Запрос 02

Данные, авторизация и интеграции

Этот запрос можно пока пропустить. Отправляйте его, если у вас уже есть что подключать: CRM, Google Таблица, платёжная система, база сотрудников, онлайн‑школа или API. Если этого пока нет — переходите сразу к запросу 03. Позже, когда нужные сервисы появятся, вы вернётесь к этому этапу и попросите Codex подключить их к проекту.

Теперь добавь логику данных и доступа. Опиши, как приложение понимает, кто зашёл: через [Telegram ID / внутренний ID сотрудника / номер телефона / почту / SSO].

Если это приложение для сотрудников: проверь пользователя по Google Sheet, CRM или базе сотрудников и покажи только его данные.

Если это магазин или открытый сервис: создай форму регистрации. Пользователь может ввести email, телефон или открыть Mini App через Telegram. Новые данные пользователя должны сохраняться в [Google Sheets / CRM / базе данных].

Если нужен код подтверждения: объясни, кто его отправляет — backend через email-сервис, SMS-сервис или Telegram-бота.

API простыми словами: API — это “мостик” между приложением и другим сервисом. Попроси Codex подключить [Google Sheets / CRM / онлайн-школу / склад / рассылки / AI], чтобы приложение само брало оттуда данные или отправляло туда новые заявки.

Оплата: если есть покупка, добавь кнопку [Купить / Оплатить / Заказать] и подключи готовую платёжную ссылку [Robokassa / Stripe / ЮKassa / CloudPayments / GetCourse].

Запрос 03

Телефонный формат, дизайн и подготовка к запуску

Проверь, что это именно формат приложения на телефон: mobile-first Telegram Mini App, а не обычная desktop-страница для ПК.

Если интерфейс получился слишком широким или похожим на сайт для компьютера, исправь его: сделай узкую мобильную ширину, крупные кнопки, удобные поля ввода и навигацию под экран телефона.

Дизайн и UX. Стиль: [современный / минималистичный / премиальный / яркий]. Цвета: [палитра]. Референсы: [ссылки или описание]. Интерфейс должен быть удобным внутри Telegram.

Используй Telegram WebApp SDK, учитывай светлую и тёмную тему Telegram, не оставляй неработающих кнопок и демонстрационных заглушек в основном сценарии.

В финале подготовь проект для GitHub и хостинга, добавь понятный README и кратко напиши: что реализовано, как запустить приложение и что логично добавить во второй версии.

Как лучше писать

Сайт тоже лучше собирать в 2–3 запроса

Запрос 01

Структура и первая версия сайта

Ты — веб-дизайнер, маркетолог и full-stack разработчик. Создай современный адаптивный сайт для [бренд / эксперт / компания / продукт].

Цель сайта: [продать / собрать заявки / объяснить услугу / записать на консультацию]. Основная аудитория: [кто эти люди].

Структура: сделай блоки [главный экран], [о продукте], [для кого], [преимущества], [кейсы / отзывы], [тарифы], [FAQ], [форма заявки].

Первый результат: собери рабочую HTML/CSS/JS‑версию, проверь мобильную адаптацию и не оставляй пустых кнопок.

Запрос 02

Данные, формы, заявки и интеграции

Этот запрос по желанию. Он нужен, если вы прямо сейчас можете дать Codex реальные данные для подключения: CRM, Google Таблицу, email‑сервис, платёжную систему, webhook, базу данных или доступ к форме заявок. Если этого пока нет — спокойно пропускайте запрос 02 и переходите к запросу 03. Формы, оплату и CRM можно добавить позже.

Теперь добавь логику данных, заявок и доступа. Опиши, как сайт понимает, кто оставил заявку или вошёл в личный кабинет: через [email / телефон / Telegram / логин и пароль / SSO].

Если это лендинг без личного кабинета: сделай форму заявки. Пользователь вводит [имя / телефон / email / Telegram / комментарий], а данные автоматически уходят в [CRM / Google Sheets / email / базу данных].

Если это сайт с личным кабинетом: добавь регистрацию и вход. Пользователь может создать аккаунт по [email / телефону / логину и паролю]. Новые данные пользователя должны сохраняться в [базе данных / CRM], а не только на странице сайта.

Если нужен код подтверждения: объясни, кто его отправляет — backend через email‑сервис, SMS‑сервис или Telegram‑бота. Не делай вид, что код появляется сам по себе.

API простыми словами: API — это “мостик” между сайтом и другим сервисом. Попроси Codex подключить [CRM / Google Sheets / онлайн‑школу / склад / рассылки / AI], чтобы сайт сам отправлял туда заявки, подтягивал статусы, открывал уроки или обновлял данные.

Оплата: если на сайте есть покупка, добавь кнопку [Купить / Оплатить / Записаться] и подключи готовую платёжную ссылку из [ЮKassa / Robokassa / CloudPayments / GetCourse / Stripe].

Если конкретные технологии пока не указаны, выбери простое и надёжное решение для первой версии и напиши, что понадобится для полноценного подключения: API‑ключ, webhook, доступ к CRM, таблице, базе данных или платёжной системе.

Запрос 03

Адаптивность, дизайн и подготовка к запуску

Проверь, что сайт корректно работает на всех основных экранах: телефон / планшет / ноутбук / большой монитор. Если элементы разъезжаются, исправь адаптивность, отступы, меню, формы и кнопки.

Если сайт получился похожим на черновик или шаблон, доработай его как готовую страницу для реального запуска: сильный первый экран, понятный оффер, заметная кнопка действия, аккуратные блоки и читаемая мобильная версия.

Дизайн и UX. Стиль: [минималистичный / премиальный / яркий / экспертный], цвета [палитра], референсы [ссылки или описание]. Сделай крупные заголовки, удобные формы, понятные CTA‑кнопки и визуальную иерархию.

SEO и техническая база. Добавь title, description, корректные заголовки, alt для изображений, favicon и базовую подготовку к аналитике. Не оставляй неработающих кнопок и демонстрационных заглушек в основном сценарии.

В финале подготовь проект для GitHub и хостинга [DigitalOcean / TimeWeb Cloud / другой доступный хостинг], добавь понятный README и кратко напиши: что реализовано, как запустить сайт, куда уходят заявки и что логично добавить во второй версии.

AI‑сервис

Что нужно упомянуть

  1. Задача: что сервис должен делать с помощью ИИ
  2. Входные данные: текст, файл, анкета, ссылка, фото
  3. Результат: что пользователь получает на выходе
  4. Пользователи: гость, клиент, админ, эксперт
  5. Ограничения: лимиты, история запросов, платный доступ
  6. Хранилище: где лежат пользователи, запросы и результаты
  7. AI: какой тип модели нужен и как формируется промпт
  8. Безопасность: какие данные нельзя отправлять в AI

AI‑сервис — это не просто “чат”. Важно описать путь: что человек вводит, что ИИ обрабатывает и какой готовый результат отдаёт.

Как лучше писать

Сначала интерфейс и логика, потом AI‑подключение

Запрос 01

Первая версия AI‑сервиса

Ты — продуктовый дизайнер и full-stack разработчик. Создай AI‑сервис [название / идея], который помогает [кому] получить [конкретный результат].

Пользователь вводит [текст / файл / ссылку / ответы анкеты], нажимает кнопку, а сервис возвращает [план / анализ / текст / подборку / отчёт / рекомендации].

Сначала сделай рабочий интерфейс: главный экран, форму ввода, экран результата, историю запросов и понятные состояния загрузки / ошибки.

Запрос 02

AI‑логика, данные, доступ и интеграции

Этот запрос можно оставить на потом. Он нужен, если вы уже готовы подключать AI API, оплату, CRM, базу данных, личный кабинет или хранение истории запросов. Если сейчас хочется просто собрать красивую первую версию сервиса — переходите к запросу 03, а интеграции добавите позже.

Теперь добавь логику данных, доступа и AI‑обработки. Опиши, как сервис понимает, кто зашёл: через [email / телефон / Telegram / Google‑аккаунт / логин и пароль / SSO].

AI‑логика: сформируй системный промпт для сервиса: какую роль выполняет AI, какой формат ответа нужен, какие ограничения соблюдать, чего не делать и как проверять качество результата.

Данные пользователя: сохраняй пользователя, запрос, дату, статус обработки и результат в [базе данных / CRM / Google Sheets для теста]. Если нужен личный кабинет, показывай человеку историю только его запросов и результатов.

API простыми словами: API — это “мостик” между AI‑сервисом и внешними системами. Попроси Codex подключить [AI API / CRM / оплату / рассылки / онлайн‑школу / базу данных], чтобы сервис отправлял запросы в AI, сохранял результаты, создавал карточки клиентов или открывал доступ после оплаты.

Безопасность: не храни AI API‑ключи в JavaScript. Вынеси их на backend или в переменные окружения. Если реальный AI API пока не подключён, сделай mock‑версию и напиши, какие ключи понадобятся во второй версии.

Оплата и лимиты: если сервис платный, добавь тарифы, бесплатный лимит [например, 3 запроса] и связь с платёжной системой [ЮKassa / Robokassa / CloudPayments / Stripe].

Запрос 03

Интерфейс, UX, безопасность и запуск

Проверь, что AI‑сервис выглядит как готовый продукт, а не просто поле ввода. Нужны: понятная главная страница, форма запроса, состояние “идёт обработка”, экран результата, история запросов и понятные ошибки.

Если интерфейс получился слишком сложным, упрости основной путь: пользователь вводит данные → нажимает кнопку → получает результат → может сохранить, скопировать или отправить его себе.

Дизайн и UX. Стиль: [современный / минималистичный / технологичный / премиальный]. Цвета: [палитра]. Референсы: [ссылки или описание]. Сделай крупные кнопки, читаемый результат и аккуратные состояния загрузки.

Безопасность и данные: отдельно напиши, какие данные пользователя сохраняются, где они хранятся, можно ли удалить историю и какие данные нельзя отправлять в AI без согласия.

В финале подготовь проект для GitHub и хостинга [DigitalOcean / TimeWeb Cloud / другой доступный хостинг], добавь README и кратко напиши: что реализовано, как запустить сервис, какие переменные окружения нужны и что логично добавить во второй версии.

Совет

Не бойтесь спрашивать Codex советы и инструкции. Например: «Какую CRM‑систему ты бы посоветовал использовать в России, чтобы потом подключить её к моему сайту?» или «Дай пошаговую инструкцию, как подключить оплату». Задавайте вопросы и следуйте инструкциям, которые Codex будет давать по вашему проекту.

Сценарий 01

Приложение для сотрудников

1

В CRM, базе или закрытой Google Таблице заранее есть сотрудник: внутренний ID, имя, должность, роль и статус.

2

При первом входе он вводит внутренний номер и одноразовый код из письма или приглашения.

3

Backend проверяет данные и привязывает к карточке проверенный числовой Telegram ID.

4

При следующих входах кабинет открывается автоматически, а информация загружается по роли сотрудника.

Не делаем: публичный список внутренних номеров, где можно выбрать любого сотрудника. Знание номера само по себе не доказывает личность.
Пример

Google Sheets может остаться вашей базой

Если у клиента уже есть Google Таблица, не обязательно сразу переносить всё в сложную CRM. В таблице можно хранить ID сотрудника, код активации, роль, статус, товары, заявки или любые другие рабочие данные.

Сотрудник открывает Mini App, вводит свой номер и код. Backend проверяет эти данные в таблице, привязывает доступ к его Telegram ID и показывает только ту информацию, которая относится к этому сотруднику.

Главная идея: вы продолжаете обновлять Google Sheets, а приложение автоматически подтягивает свежие данные для каждого сотрудника, товара или заявки.
Про данные: Google Sheets — хороший быстрый вариант, чтобы протестировать идею и показать клиенту MVP. Но если в проекте есть персональные данные, в идеале лучше использовать защищённую CRM, базу данных или сервис, где можно настроить права доступа, журнал действий и хранение данных по правилам.
Важно по РФ: если вы работаете с персональными данными пользователей из России, учитывайте требования Федерального закона № 152‑ФЗ «О персональных данных». Для хранения и учета данных рекомендуется использовать проверенные CRM‑системы и сервисы, соответствующие требованиям российского законодательства, а не хранить персональные данные напрямую в зарубежных сервисах.
Важно настроить отдельно

Чтобы изменения из таблицы появлялись в приложении

Само по себе приложение не узнаёт, что вы изменили данные в Google Sheets. Таблицу и Mini App нужно один раз связать через Google Apps Script или backend.

После настройки вы продолжаете работать в привычной таблице: меняете должность, смены, документы, товары или статусы — а приложение получает актуальную информацию при открытии, обновлении или автоматически в режиме реального времени.

Что написать Codex:

«Помоги связать мою Google Таблицу с Telegram Mini App. Дай подробную пошаговую инструкцию и настрой приложение так, чтобы изменения из таблицы автоматически появлялись в приложении в режиме реального времени».

Codex уточнит структуру таблицы, подскажет, какие доступы нужны, поможет создать связку и объяснит, куда добавить полученный код.

Сценарий 02

Магазин или открытый сервис

1

Нужно создать форму регистрации: пользователь вводит email, телефон или открывает Mini App через Telegram.

2

Backend создаёт нового пользователя: сохраняет email, телефон, Telegram ID, дату регистрации и статус клиента.

3

Эти данные можно автоматически отправлять в Google Таблицу, CRM или базу данных — чтобы менеджер сразу видел нового пользователя.

4

Если нужен код подтверждения, его отправляет backend через email-сервис, SMS-сервис или Telegram-бота. Пользователь вводит код и получает доступ.

Важно: Google Таблица или CRM могут быть “приёмником” данных. Например: человек зарегистрировался по email — его email, имя, телефон, дата регистрации и источник сразу попали в таблицу или карточку CRM.
Сценарий 03

Онлайн-школа или закрытый клуб

1

Ученик получает доступ после оплаты, от преподавателя или от администратора: код, ссылку-приглашение или QR-код.

2

При первом входе приложение проверяет код и привязывает доступ к Telegram ID конкретного ученика.

3

Дальше ученик видит только свои уроки, материалы, домашние задания, прогресс или закрытые разделы клуба.

4

Если оплата проходит в GetCourse, Robokassa, Stripe или другой системе, приложение может открывать готовую ссылку оплаты, а доступ выдавать после подтверждения.

Самый простой старт: кнопка «Оплатить» ведёт на готовую страницу оплаты, а код доступа выдаётся после покупки.
Сайт

Как возможна авторизация на сайте

1

Простой сайт без кабинета: авторизация не нужна. Пользователь оставляет заявку, нажимает кнопку оплаты или пишет в Telegram. Данные из формы уходят в CRM, Google Sheets, email или базу данных.

2

Сайт с личным кабинетом: пользователь регистрируется по email, телефону или через соцсети. Backend создаёт аккаунт и хранит профиль в базе данных или CRM.

3

Коды подтверждения: их отправляет не сам сайт, а подключённый сервис: email‑сервис, SMS‑провайдер, Telegram‑бот или готовая auth‑платформа.

4

Куда уходят данные: email, телефон, имя, дата регистрации, заявки и покупки сохраняются в базе данных, CRM или таблице для теста. Для рабочего продукта лучше использовать базу данных или CRM.

Что нужно создать или подключить: backend для обработки регистрации, базу данных или CRM для хранения пользователей, сервис отправки кодов, а при оплате — платёжную систему.

AI‑сервис

Как авторизуется пользователь в AI‑сервисе

1

Гостевой режим: человек может сделать 1–3 тестовых запроса без регистрации. Сервис хранит только временную сессию в браузере или на backend.

2

Личный кабинет: пользователь входит по email, телефону, Telegram или Google‑аккаунту. Так можно хранить историю запросов, результаты, тариф и лимиты.

3

Где всё хранится: сервису нужно место, куда он будет записывать пользователей и их результаты. Например: кто зарегистрировался, какой запрос отправил, какой ответ получил, оплатил ли доступ. Для теста можно использовать Google Sheets, но для нормального проекта лучше CRM или база данных.

4

Как работает AI: человек вводит текст на сайте, а сайт отправляет его в AI‑сервис и получает готовый ответ. Для этого нужен “ключ доступа” к AI. Его нельзя оставлять прямо в коде сайта, потому что его могут увидеть другие люди. Поэтому ключ хранится в закрытой части проекта — на backend.

Что нужно создать или подключить: backend, базу данных, AI API, систему лимитов, оплату при платном доступе и понятную политику хранения пользовательских запросов.
Аккуратно с персональными данными: если пользователь вводит документы, телефоны, медицинские, финансовые или другие чувствительные данные, заранее решите, что можно отправлять в AI‑сервис, что нужно обезличивать и где хранить результаты.
